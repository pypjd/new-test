# 旅行轨迹记录软件（详细 README）

## 一、这份代码是做什么的

这是一个基于 **React + TypeScript + Vite + Leaflet + Express** 开发的旅行轨迹记录工具。

它的核心用途不是单纯“看地图”，而是把一次旅行拆成：

- **旅程（Trip）**
- **日期（Day）**
- **路段（Segment）**

然后围绕每一段路程记录：

- 起点 / 终点
- 途经点
- 路线类型（驾车 / 骑行）
- 路线策略（当前代码中为“高速优先 / 避免收费”）
- 实际规划出的轨迹折线
- 路段里程、当日总里程、旅程总里程

同时，项目已经支持两类工作区：

- **复盘（review）**：适合记录已经走过的路线
- **规划（plan）**：适合提前做行程规划

### 1. 为什么一定要有 Key

因为下面这些功能都依赖高德：

- 地名联想
- 驾车路径规划
- 骑行路径规划

### 2. 推荐的配置方式

项目根目录提供了 `.env.example`，推荐复制为 `.env.local` 后填写：

```bash
AMAP_WEB_API_KEY=你的高德Web服务Key
BACKEND_PORT=3001
VITE_BACKEND_BASE_URL=http://localhost:3001
VITE_APP_MODE=normal
```

后端会按下面优先级读取 Key：

1. `AMAP_WEB_API_KEY`
2. `AMAP_WEB_KEY`
3. `AMAP_KEY`

## 六、项目当前支持的核心功能

根据当前代码，已经实现的主要功能包括：

### 1. 旅程管理

- 新增旅程
- 编辑旅程名称、开始日期、结束日期
- 删除旅程
- 旅程上下移动排序
- 按工作区区分为“复盘 / 规划”

### 2. 路段管理

- 为某个旅程、某一天新增路段
- 编辑路段名称
- 修改路段所属日期
- 删除路段
- 在同一天内对路段上移 / 下移

### 3. 起终点与途经点编辑

- 起点、终点支持输入地名并联想
- 途经点支持新增、删除、上移、下移
- 支持在地图中定位某个途经点
- 支持把搜索结果写回坐标

### 4. 地图轨迹规划与展示

- 使用高德进行驾车路线规划
- 支持骑行路线规划
- 当前代码中的路线策略为：
  - 高速优先
  - 避免收费
- 地图支持单条路段、单日全部路段、单旅程全部路段、全部旅程总览

### 5. 距离统计

- 单条轨迹里程
- 当日总里程
- 旅程总里程
- 如果高德没直接返回里程，会回退为根据折线点计算近似距离

### 6. 本地持久化

- 旅程结构保存在 `localStorage`
- 大体积轨迹点保存在 `IndexedDB`
- 重启浏览器后仍可读取

### 7. 只读演示模式

- 从静态 JSON 读取演示数据
- 禁止写操作
- 适合部署成展示网页

## 八、代码详细分析

下面这一部分重点不是“怎么运行”，而是“这份代码本身是怎么组织的”。

---

### 1. 前端总控：`src/App.tsx`

这是整个前端的总调度中心。

它负责做几件关键的事：

#### （1）维护应用级状态

主要状态包括：

- 当前工作区：`review` / `plan`
- 当前筛选条件：`tripId / dayId / segmentId`
- 当前是否正在编辑某条路段
- 当前是否正在编辑途经点
- 当前是否正在编辑起终点
- 当前是否打开旅程管理弹窗

#### （2）串联几个核心 Hook

`App.tsx` 并没有把所有逻辑都堆在自己内部，而是把不同职责拆给不同 Hook：

- `useTripReviewState()`：负责总数据加载与保存
- `useFilteredSegments()`：根据筛选条件计算当前要展示的路段
- `useRouteCacheHydration()`：把 IndexedDB 中缓存的路线点重新注入到内存状态
- `useTripManager()`：负责新增、删除、更新旅程和路段
- `useSegmentEditing()`：负责路段详情编辑、途经点编辑、起终点编辑

这种写法的优点是：

- `App.tsx` 虽然大，但仍然是“编排层”，不是所有业务细节都塞在一个文件里
- 后续继续扩展时，可以继续沿着 Hook 分层的方式迭代

#### （3）决定页面三大区域的数据来源

页面基本可以看成三块：

1. `TripEditor`：新增旅程、新增路段
2. `FilterPanel`：筛选区
3. `MapPlaceholder + MapPanel`：路段详情区 + 真正地图区

`App.tsx` 负责把当前工作区、当前筛选、当前活跃路段、当前统计结果统一算好，再传给子组件。

#### （4）计算总览信息

比如：

- 当前旅程总里程
- 当前日期总里程
- 当前地图上显示的路段数
- 当前地图信息栏文字

这些都在 `App.tsx` 里统一用 `useMemo` 计算，避免每个子组件重复算一遍。

---

### 2. 领域模型：`src/types/trip.ts`

这个文件定义了整套数据模型，是整个项目最核心的“数据约定”。

主要结构如下：

#### `TripReview`

根对象，只有一个字段：

- `trips: Trip[]`

#### `Trip`

表示一次完整旅程，包含：

- `id`
- `title`
- `category`：`review` 或 `plan`
- `order`
- `startDate`
- `endDate`
- `days`

#### `TripDay`

表示旅程中的一天，包含：

- `id`
- `date`
- `routeSegments`

#### `RouteSegment`

表示一段路线，是最重要的数据对象，包含：

- 路段名称
- 起点 / 终点文本
- 起终点坐标
- 途经点列表
- 路线类型
- 路线偏好
- 轨迹点 `points`
- 里程 `distanceMeters`
- 路线签名 `routeBuildKey`

这个模型设计得比较合理，因为它把：

- “用户输入的文本”
- “解析后的坐标”
- “真正生成出来的折线”

三层信息分开保存了。后续不管是重新规划、缓存校验、导出演示数据，都会比较方便。

---

### 3. 数据持久化：`src/services/tripStorage.ts`

这个文件负责 `localStorage` 的读写。

它做得比较好的地方有三点：

#### （1）统一入口

组件本身不直接操作 `localStorage`，而是统一走：

- `loadTripReview()`
- `saveTripReview()`
- `toPersistedTripReview()`

这样后面如果你想换成别的存储方式，不需要全项目到处改。

#### （2）做了旧数据兼容

代码里兼容了旧版本的 `viaPointsText`，会自动把它转换成新的 `waypoints` 结构。

这说明这个项目已经不是“从零一次成型”，而是在迭代过程中考虑了历史数据迁移。

#### （3）把大轨迹点从 localStorage 中剥离出去

当前代码会故意把 `points` 设为 `undefined` 再保存到 `localStorage`。

原因很明确：

- 路线折线点很多
- 全部塞进 `localStorage` 很容易超限
- 所以把结构化元数据留在 `localStorage`
- 把大体积折线单独放进 `IndexedDB`

这是这份代码里一个很关键、也比较成熟的设计点。

---

### 4. 轨迹缓存：`src/services/routeCacheDb.ts`

这个文件负责 `IndexedDB` 层面的路线缓存。

它实现了：

- `saveSegmentRouteCache()`：保存某段路线折线
- `getSegmentRouteCache()`：读取某段路线折线
- `deleteSegmentRouteCache()`：删除某段路线缓存
- `clearAllRouteCache()`：清空全部缓存

这里的设计目的非常明确：

- **减少高德 API 调用次数**
- **减少每次打开页面时重新规划所有轨迹的开销**
- **避免浏览器 localStorage 体积爆掉**

缓存记录里不仅保存 `segmentId`，还保存 `routeBuildKey`。

这很重要，因为它保证了：

- 不是“只要 segmentId 相同就直接复用”
- 而是“只有当前起终点 / 途经点 / 路线类型 / 路线策略与缓存时一致，才复用旧轨迹”

这就避免了“用户改了路线条件，但地图还显示旧线”的错误。

---

### 5. 应用模式：`src/config/appMode.ts`

这个文件很小，但作用很关键。

它定义了两个模式：

- `normal`
- `readonly-demo`

也就是：

- 平时本地正常编辑模式
- 只读展示模式

后续如果你继续升级项目，这里还可以扩展出更多模式，比如：

- `public-demo`
- `self-hosted`
- `desktop`

目前这层模式隔离已经有雏形了。

---

### 6. 数据总状态 Hook：`src/hooks/useTripReviewState.ts`

这个 Hook 负责“应用一启动，数据从哪里来”。

#### 在 normal 模式下

- 直接从 `localStorage` 读取
- 并在状态变化后自动保存

#### 在 readonly-demo 模式下

- 不读 `localStorage`
- 转而读取 `public/demo-data/manifest.json`
- 再把多个分片 JSON 合并成一个 `TripReview`

这个设计说明你的项目已经开始区分：

- **编辑型数据源**
- **演示型数据源**

对于后续上线展示版，这是很重要的一步。

---

### 7. 路线缓存回填 Hook：`src/hooks/useRouteCacheHydration.ts`

这个 Hook 的作用是：

- 应用加载后
- 逐个扫描当前旅程里的路段
- 去 IndexedDB 查有没有缓存折线
- 如果缓存的 `routeBuildKey` 与当前路段匹配
- 就把 `points` 回填到运行时状态里

这一步实际上解决了一个非常现实的问题：

> 旅程结构存在 `localStorage`，轨迹点存在 IndexedDB，页面启动时如何把两边拼回完整状态？

这个 Hook 就是在做这件事。

从工程角度看，这属于“存储层拆分后必要的状态重建逻辑”，做得是对的。

---

### 8. 筛选计算 Hook：`src/hooks/useFilteredSegments.ts`

这个 Hook 很纯粹，只做一件事：

- 根据 `tripId / dayId / segmentId` 三级筛选条件
- 输出当前应该展示哪些 `RouteSegment`

它的好处是：

- 逻辑简单
- 可预测
- 容易调试
- 不和 UI 混在一起

对于这类列表 / 筛选型页面，这种“纯计算 Hook”非常值得保留。

---

### 9. 旅程与路段管理 Hook：`src/hooks/useTripManager.ts`

这是当前前端业务逻辑里最核心的文件之一。

它处理的能力很多，包括：

- 查找某条路段所在的旅程 / 日期位置
- 新增旅程
- 新增路段
- 更新路段
- 更新路段元信息
- 删除路段
- 删除旅程
- 旅程上下移动
- 路段上下移动
- 只读模式写入拦截

#### 这个 Hook 的优点

##### （1）写操作集中

所有“改数据”的动作，基本都集中在这里，而不是分散到每个组件内部。

##### （2）删除时会同步清缓存

删除路段或旅程时，会调用 `deleteSegmentRouteCache()` 去清掉对应路线缓存。

这一点很好，因为它避免了无效旧缓存残留。

##### （3）路段跨日期移动逻辑已经考虑到了

`updateSegmentMeta()` 里，如果用户把路段日期改成另一天：

- 会从原来的 day 中移除
- 再插入目标日期对应的 day
- 如果目标日期不存在，会自动新建 day

这说明数据结构的改动逻辑已经比较完整，不是简单“改个字段”而已。

#### 这个 Hook 的不足

它现在已经比较长了，继续增加业务后，建议以后再拆成几个方向：

- `useTripCrud`
- `useSegmentCrud`
- `useTripOrdering`
- `useReadonlyGuard`

这样会更利于长期维护。

---

### 10. 路段编辑 Hook：`src/hooks/useSegmentEditing.ts`

这个 Hook 更偏向“当前活动路段的编辑状态管理”。

它主要负责：

- 当前活动路段日期
- 当前展示的途经点
- 当前选中的途经点
- 起终点草稿
- 路段名称 / 日期草稿
- 进入编辑 / 保存编辑 / 取消编辑

它把“编辑中的临时草稿”和“真正保存后的数据”区分开了，这一点很重要。

否则一边输入、一边直接改主状态，很容易出现：

- 输入中断
- 联想组件闪烁
- 取消编辑无法回滚
- 数据半改状态

你这份代码在这里已经有比较清晰的草稿层。

---

### 11. 地图服务层：`src/services/amap.ts`

这是整个项目另一个非常关键的核心文件。

它负责把前端对“地名联想”和“路线规划”的需求统一封装起来。

#### 它做了哪些事

##### （1）输入提示缓存

- 使用内存 Map 做 LRU/TTL 式缓存
- 最多缓存 200 条
- 默认 10 分钟过期

##### （2）输入提示双通道查询

`searchAmapInputTips()` 会并行请求：

- 行政区联想（city）
- 普通 POI 联想（poi）

然后合并、去重，并把行政区优先显示。

如果启用了城市范围锁定且结果太少，还会再发一次范围外补充查询。

这套逻辑说明你并不是只做了“最简单的输入框请求”，而是已经在优化实际选点体验。

##### （3）路线规划队列限流

这里做了一个很有价值的优化：

- 并发数限制为 2
- 每次请求前延迟 200ms
- 命中限流错误后暂停 3 秒再继续

这能显著减少大量路段同时规划时把高德接口打爆的风险。

##### （4）进行请求去重

对相同路线条件的请求，会做 in-flight 去重，避免同一条路线被短时间重复请求多次。

##### （5）内存缓存路线

规划过的路线会先存在前端内存 Map 里，如果参数没变，可以立即复用。

##### （6）支持驾车与骑行两类规划

- 驾车：走 `/api/amap/direction`
- 骑行：走 `/api/amap/cycling-direction`

其中骑行模式会把多段 waypoint 路线拆成若干小段逐段规划，再拼接 polyline。

#### 这一层的整体评价

这不是“演示级别”的接口调用写法，而是已经有了一定真实使用场景下的性能和配额意识。

这是当前项目里比较成熟的一部分。

### 12. 地图组件：`src/components/MapPanel.tsx`

这是前端最复杂的组件之一，负责真正的地图渲染与轨迹规划展示。

它主要做的事包括：

#### （1）把当前筛选到的路段转成地图轨迹

如果某段路线已经有可复用的缓存折线，就直接展示。

如果没有缓存：

- 在普通模式下会按需调用高德规划
- 在只读模式或全局总览限制下，会优先只展示已缓存轨迹

#### （2）自动补解析起终点坐标

如果路段只有起终点名称，没有坐标，会先用输入提示服务尝试解析坐标后再规划。

#### （3）失败时降级为直线

如果高德没有成功返回 polyline，组件会用起点-途经点-终点直接连成 fallback 折线。

这比“空白不显示”体验好很多。

#### （4）保存轨迹缓存

规划成功后的折线会写入 IndexedDB。

#### （5）总览模式自动抽稀

在总览视图里，会对过长折线进行 downsample，降低渲染压力。

这也是一个很实际的性能优化。

#### （6）地图交互辅助控制器

文件中还写了几个控制器式小组件：

- `ViewportController`：根据轨迹自动 fitBounds
- `MapResizeController`：处理尺寸变化导致的地图刷新问题
- `WaypointFocusController`：聚焦到选中的途经点

这说明你在处理 Leaflet 细节时，已经不是“只会画图层”，而是开始处理真实交互问题了。

### 23. Vite 配置：`vite.config.ts`

Vite 配置里最关键的是：

- 开发环境下把 `/api` 代理到独立后端
- preview 预览模式同样代理 `/api`

这样前端代码里始终只需要请求 `/api/...`，不需要到处写死完整后端地址。

这种做法能明显减少前后端切换环境时的混乱。
